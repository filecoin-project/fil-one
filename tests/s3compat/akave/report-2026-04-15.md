## Root causes sorted by impact

### 1. No ACL / authorization enforcement (~50-60 tests)

The gateway does not enforce access control at any level. Every `alt_client`
(second user) operation succeeds where it should be denied.

**Symptoms**:

- All `test_access_bucket_*` tests: `check_access_denied` always raises
  `ClientError not raised`.
- ACL canned-policy tests (`public-read`, `authenticated-read`, etc.): setting
  them has no effect; the grants list always contains only FULL_CONTROL for the
  owner.
- Bucket-policy enforcement tests: `alt_client` can list, read, write, copy,
  and create multipart uploads regardless of the policy.
- Encryption-policy enforcement tests: bucket policies that require encryption
  are ignored.
- `PutBucketAcl` with grant structures fails because the gateway cannot
  serialize them.

**Affected categories**: s3_core, bucket_policy, encryption, tagging, copy

### 2. Versioning not implemented (~40-50 tests)

`PutBucketVersioning` returns an error, which cascades through every test that
depends on versioning.

**Symptoms**:

- All `test_versioning_*` tests fail immediately on `PutBucketVersioning`.
- All delete-marker tests (3 tests) fail.
- Versioned-copy tests fail.
- All ~20 object-lock tests fail (object lock requires versioning).
- `GetBucketVersioning` on a fresh bucket returns `Enabled` instead of no
  status, so `check_versioning(bucket_name, None)` also fails.

This root cause is also the main driver of **cascading teardown failures**
(~30 error-outcome tests toward the end of the run): object-lock tests create
locked objects that cannot be deleted, and subsequent tests fail in setup
because `nuke_bucket` -> `DeleteBucket` cannot remove the poisoned bucket.

**Affected categories**: s3_core, delete_marker, copy

### 3. Gateway injects internal metadata and tags (~25-30 tests)

Every object stored through the gateway acquires four extra user-metadata
entries (`Network-File-Name`, `Network-Root-Cid`, `Network-Processed`,
`Network-State`) and four corresponding object tags (`network-root-cid`,
`network-file-name`, `network-state`, `network-processed`).

**Symptoms**:

- Metadata equality checks fail: e.g. `assert got == {}` returns
  `{'Network-File-Name': '...', 'Network-Processed': '0', ...}`.
- Copy-metadata tests fail because extra keys appear and user-supplied key
  casing changes (`key1` -> `Key1`).
- All tagging tests fail: `TagSet` always contains 4 extra system tags;
  tag-count assertions, tag-value assertions, and tag-deletion tests all break.
  After `DeleteObjectTagging`, 4 system tags remain.
- Tag-conditional bucket-policy tests fail because the tag set never matches
  expected values.

**Affected categories**: s3_core, tagging, copy, encryption

### 4. POST Object (form upload) not supported (~25 tests)

The `POST /{bucket}` form-based upload endpoint returns 403 Forbidden for every
request, regardless of authentication.

**Symptoms**:

- Every `test_post_object_*` test fails with `assert 403 == 204` (or 201, 200,
  400 depending on the expected outcome).
- Includes SSE-C POST, SSE-KMS POST, and tagging POST tests.

**Affected categories**: s3_core, encryption, tagging

### 5. ListObjects pagination and encoding broken (~20-25 tests)

Multiple interrelated bugs in the ListObjects / ListObjectsV2 implementation.

**Sub-issues**:

| Issue | Example |
| ----- | ------- |
| `MaxKeys` ignored | `MaxKeys=0` returns all objects; `MaxKeys=1` does not limit |
| Marker/ContinuationToken pagination wrong | Marker-based listing skips too many or too few keys |
| `NextMarker` returns base64 internal value | `'Ym9vL2Jhci9WMQ=='` instead of `'asdf'` |
| URL-encoding leaks into SDK values | Delimiter/Prefix returned as `%2F` instead of `/` when `encoding-type=url` |
| Missing response fields | `Prefix` absent when no prefix set; `Delimiter` present when it should be omitted for empty/none |
| Sort order incorrect | Keys with special characters return in wrong order vs. lexicographic; paginated prefix listing returns wrong prefixes |

**Affected categories**: s3_core, list_objects_v2

### 6. Encryption handling incomplete (~20 tests)

The gateway accepts all encryption headers silently without acting on them.

**Symptoms**:

- SSE-C ignored entirely: `HeadObject` / `GetObject` without the customer key
  succeeds when it should return 400.
- Invalid/missing SSE-C key parameters accepted without error.
- SSE-KMS misreported: HEAD response returns `AES256` instead of `aws:kms`.
- No encryption-conflict detection: conflicting SSE-C / SSE-S3 / SSE-KMS
  parameters all accepted.

**Affected categories**: encryption, sse_s3

### 7. Multipart upload validation missing (~15 tests)

`CompleteMultipartUpload` does not validate its input, and some valid completions
also fail.

**Symptoms**:

- Empty parts list accepted (should return error).
- Missing or incorrect ETags accepted (should return `InvalidPart`).
- Part-size minimums not enforced (should return `EntityTooSmall`).
- `CompleteMultipartUpload` fails on legitimate multi-part completions.
- `ListMultipartUploads` shows an internal key format (`multipart1/V1` instead
  of `multipart1`).
- Second concurrent multipart upload on the same bucket fails.

**Affected categories**: s3_core, copy

### 8. Special characters in object keys (~10 tests)

`PutObject` fails for keys containing URL-sensitive characters.

**Symptoms**:

- Keys with `%` (`b%ar`, `b%az`) are rejected or misinterpreted.
- Keys ending with `/` (`asdf/`) fail.
- Keys with special ASCII characters that have URL-encoding significance fail.

This likely indicates the gateway URL-decodes or rejects keys that contain
characters it interprets as path separators or encoding sequences.

**Affected categories**: s3_core, list_objects_v2, copy

### 9. Miscellaneous missing response fields and wrong error codes (~10 tests)

A collection of smaller gaps:

- **Content-Type not preserved**: uploading with `ContentType='foo/bar'` returns
  `application/octet-stream`.
- **Owner missing from ListObjects Contents**: `KeyError: 'Owner'`.
- **ETag missing from error-response headers**.
- **`x-amz-tagging-count` missing from HEAD responses**.
- **`ContentEncoding` not preserved**.
- **Wrong error codes**: `InvalidArgument` returned instead of
  `InvalidBucketName` for short bucket names.
- **`x-amz-request-id` missing from error responses**.
- **Bucket names with periods rejected**: `aaa.111` fails to create.
- **CreateBucket idempotency wrong**: re-creating an existing bucket succeeds
  instead of returning `BucketAlreadyOwnedByYou`.
