## Detailed Feature Analysis

### Fully Supported Features

**Object Operations**

- PUT, GET, HEAD, DELETE objects work reliably
- Copy objects within and across buckets
- Range requests for partial downloads
- ETags and conditional requests (If-Match/If-None-Match)

**Multi-part Uploads**

- Create, upload parts, complete, and abort multipart uploads
- List in-progress uploads
- Suitable for large file uploads
- _Caveat: Some edge-case validations (empty uploads, small parts) differ from S3 spec_

**Authentication & Addressing**

- AWS Signature V4 authentication fully functional
- Path-style addressing works throughout
- Presigned GET URLs for time-limited access

**Metadata**

- Custom `x-amz-meta-*` headers preserved on objects
- Content-Type, Content-Encoding, Content-Disposition supported
- Additional network metadata included (by design)

**Bucket Management**

- Create, list, and delete buckets
- _Caveat: Bucket naming error codes differ; recreation of existing buckets allowed_

### Partially Supported Features

**Server-Side Encryption**

- Object payloads are encrypted at rest (SSE-S3, SSE-C, SSE-KMS)
- _Gap: Response headers (`x-amz-server-side-encryption`) not returned_
- _Gap: Bucket-level default encryption configuration API not implemented_

**Object Listing**

- ListObjects V1 and V2 work for normal use cases
- Prefix filtering and delimiters functional
- _Gap: `MaxKeys=0` returns all objects instead of none_
- _Gap: Pagination (Marker/ContinuationToken) may return incorrect results for large buckets_

**Presigned URLs**

| Operation                 | Status            | Notes                                                    |
| ------------------------- | ----------------- | -------------------------------------------------------- |
| GET Object                | **Supported**     | Download objects via presigned URL                       |
| PUT Object                | **Unknown**       | Tests only cover PUT with ACL headers, which fails (400) |
| PUT Object with ACL       | **Not Supported** | Returns 400                                              |
| HEAD Object               | **Not Tested**    | No test coverage                                         |
| DELETE Object             | **Not Tested**    | No test coverage                                         |
| List Objects              | **Not Tested**    | No test coverage                                         |
| POST Object (form upload) | **Not Supported** | Returns 403                                              |

- Presigned GET URLs work correctly for basic downloads
- _Gap: Presigned PUT with ACL headers returns 400_
- _Gap: Presigned POST (browser form uploads) returns 403_
- _Gap: Other presigned operations (HEAD, DELETE, LIST) not tested_

> **Test Coverage Gap**: The Ceph S3 test suite does not include tests for basic presigned PUT URLs without ACL headers, nor for presigned HEAD, DELETE, or LIST operations. Additional testing is recommended to verify these operations work correctly with Akave S3.

### Not Supported Features

**Versioning**

- `put_bucket_versioning` fails
- Cannot enable/suspend versioning on buckets
- Version-aware operations non-functional

**Object Lock & Retention**

- Object Lock configuration accepted but not enforced correctly
- Legal holds and retention periods not functional
- `BypassGovernanceRetention` for cleanup doesn't work
- _Impact: Cannot meet compliance requirements (SEC 17a-4, WORM storage)_

**POST Object (Browser Uploads)**

- All POST-based uploads return HTTP 403
- Affects web applications with direct browser-to-storage uploads

## Failures Grouped by Root Cause

### 1. Versioning Not Supported (~50+ tests)

**Symptom**: `put_bucket_versioning` fails with an error

**Affected tests**: All versioning-related operations including:

- `test_versioning_bucket_create_suspend`
- `test_versioning_obj_create_read_remove`
- `test_versioning_multi_object_delete`
- `test_versioned_object_acl`
- `test_delete_marker_versioned`
- `test_copy_object_versioned_*`
- Multiple `conditional_write` tests

**Analysis**: Akave S3 either doesn't support versioning or has it enabled by default (status shows `'Enabled'` when `None` is expected). Tests trying to enable versioning with `put_bucket_versioning` fail.

---

### 2. MaxKeys/Pagination Behavior Incorrect (~8 tests)

**Symptom**: `MaxKeys=0` returns all objects instead of none; `Marker` pagination returns wrong results

**Affected tests**:

- `test_bucket_list_maxkeys_zero` - Expected `[]`, got all 4 keys
- `test_bucket_listv2_maxkeys_zero` - Same issue
- `test_bucket_list_maxkeys_one` - Second page includes objects already seen
- `test_bucket_listv2_maxkeys_one` - Same issue
- `test_bucket_list_marker_not_in_list` - Missing 'foo' from result
- `test_bucket_listv2_continuationtoken` - Wrong keys returned

**Analysis**: Akave doesn't properly respect `MaxKeys=0` (should return empty list) and pagination (Marker/ContinuationToken) doesn't correctly exclude already-seen items.

---

### 3. URL Encoding in Response Fields (~10 tests)

**Symptom**: Delimiter/Prefix returned as URL-encoded (`%2F` instead of `/`)

**Affected tests**:

- `test_bucket_listv2_encoding_basic` - `'%2F'` != `'/'`
- `test_bucket_list_encoding_basic` - Same
- `test_bucket_list_prefix_basic` - `'foo%2F'` != `'foo/'`
- `test_bucket_list_prefix_unreadable` - `'%0A'` != `'\n'`
- `test_bucket_list_prefix_delimiter_basic`

**Analysis**: When `EncodingType=url` is requested, Akave URL-encodes the `Delimiter` and `Prefix` fields in responses. S3 spec expects these to be returned as-is.

---

### 4. POST Object Upload Not Supported (~30 tests)

**Symptom**: All POST object operations return HTTP 403

**Affected tests**: All `test_post_object_*` tests:

- `test_post_object_anonymous_request`
- `test_post_object_authenticated_request`
- `test_post_object_set_success_code`
- `test_post_object_upload_larger_than_chunk`
- All encryption POST tests

**Analysis**: Akave S3 does not support browser-based POST uploads (form-based multipart uploads). All POST requests return 403 Forbidden.

### 5. Internal Metadata Leaking to Tags/Metadata (~25 tests)

**Symptom**: Internal Akave metadata appears in object tags and metadata

**Affected tests**:

- `test_get_obj_tagging` - Returns extra tags: `network-root-cid`, `network-file-name`, `network-state`, `network-processed`
- `test_put_obj_with_tags` - Same extra tags
- `test_object_metadata_replaced_on_put` - Metadata contains `Network-File-Name`, `Network-Processed`, `Network-Root-Cid`
- `test_delete_tags_obj_public` - Tags not actually deleted, 4 remain

**Analysis**: Akave injects internal network-related metadata (`Network-*`) into object metadata and tags. These should be hidden from the S3 API.

## Top 5 Gaps Most Likely to Impact Users

### 1. Pagination and Listing Behavior (MaxKeys, Marker, ContinuationToken)

**User Impact**: Any application that lists objects in large buckets will see incorrect results. AWS SDKs and CLI tools rely on proper pagination to iterate through buckets with thousands of objects. When `MaxKeys=0` returns all objects or markers don't work correctly, users may experience:

- Incomplete listings
- Duplicate objects in results
- Memory issues from unexpectedly large responses
- Broken backup/sync tools (aws s3 sync, rclone)

### 2. Bucket Policies Not Enforced

**User Impact**: Organizations that need to share data across accounts or grant specific permissions cannot rely on bucket policies. This breaks common patterns like:

- Cross-account access for data sharing
- Granting read-only access to analytics services
- Restricting access to specific IP ranges or VPCs
- Implementing least-privilege access controls

### 3. Versioning and Object Lock Not Functional

**User Impact**: Applications requiring data protection, compliance, or point-in-time recovery cannot use Akave. This breaks critical use cases:

- Ransomware protection through immutable backups
- Regulatory compliance (SEC 17a-4, FINRA, HIPAA) requiring WORM storage
- Accidental deletion recovery
- Audit trails with version history
- Data retention policies with legal holds

### 4. POST Object Upload Not Supported (Browser-Based Uploads)

**User Impact**: Web applications that allow direct browser-to-S3 uploads cannot use Akave. This affects:

- File upload widgets in web applications
- Presigned POST URLs for direct uploads
- Applications avoiding server-side upload proxying for performance
- Mobile apps using similar browser-based upload flows

### 5. Encryption Response Headers Missing

**User Impact**: Applications that verify encryption status of objects will fail or report incorrect security posture. This affects:

- Compliance tools that audit encryption at rest
- Applications that conditionally handle encrypted vs unencrypted objects
- Security scanning tools
- Backup solutions that preserve encryption metadata
