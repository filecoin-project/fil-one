## Detailed Feature Analysis

### Fully Supported Features

**Object Operations**

- PUT, GET, HEAD, DELETE objects work reliably
- Range requests for partial downloads
- ETags and conditional requests (If-Match/If-None-Match)
- Content-Type, Content-Encoding, Content-Disposition supported

**Multi-part Uploads**

- Create, upload parts, complete, and abort multipart uploads
- List in-progress uploads
- Suitable for large file uploads
- _Caveat: Metadata keys are converted to Title-Case (see below)_

**Authentication & Addressing**

- AWS Signature V4 authentication fully functional
- Path-style addressing works throughout
- Presigned PUT URLs for uploads work correctly
- Presigned HEAD and LIST operations work correctly

**Bucket Management**

- Create, list, and delete buckets
- Bucket versioning can be enabled
- Lifecycle configuration supported

**Encryption**

- SSE-S3 (server-side encryption with S3-managed keys) works
- SSE-C (customer-provided keys) works
- SSE-KMS (AWS KMS keys) works
- _Caveat: Metadata key case issue affects multipart uploads with encryption_

### Partially Supported Features

**Custom Metadata (x-amz-meta-\*)**

- Custom metadata is preserved on objects
- **Gap: Metadata keys are not normalized to lowercase**
  - HTTP header names are case-insensitive; boto3 (and therefore ceph-s3-tests) normalizes `x-amz-meta-*` keys to lowercase in the `Metadata` dict
  - FTH returns Title-Case keys (e.g. `Foo` for `x-amz-meta-foo`), which fails the test expectation
  - Affects: Copy operations, multipart uploads, encryption tests

**Presigned URLs**

| Operation            | Status            | Notes                                   |
| -------------------- | ----------------- | --------------------------------------- |
| PUT Object           | **Supported**     | Upload objects via presigned URL        |
| HEAD Object          | **Supported**     | Check object metadata via presigned URL |
| LIST Objects         | **Supported**     | List bucket contents via presigned URL  |
| GET Object           | **Not Supported** | Returns HTTP 401 Unauthorized           |
| DELETE Object        | **Not Supported** | Returns HTTP 401 Unauthorized           |
| GET Object Retention | **Not Supported** | Returns HTTP 401 Unauthorized           |

**Object Lock & Retention**

- Object Lock configuration can be set and retrieved
- Legal holds and retention periods functional
- **Gap: Returns HTTP 502 instead of 409 for invalid bucket configurations**
- **Gap: Error code `NoSuchBucket` returned instead of `ObjectLockConfigurationNotFoundError`**

**Lifecycle Configuration**

- Lifecycle rules can be set and retrieved
- **Gap: Returns `NoSuchBucket` instead of `NoSuchLifecycleConfiguration` after deletion**

### Not Supported Features

**ACLs (Access Control Lists)**

S3 "canned ACLs" are predefined access policies you can apply to buckets/objects:

- `private` - owner only (default)
- `public-read` - anyone can read
- `public-read-write` - anyone can read and write
- `authenticated-read` - any authenticated AWS user can read
- `bucket-owner-read` - bucket owner can read objects created by others
- `bucket-owner-full-control` - bucket owner has full control of objects

FTH behavior:

- **API accepts these ACLs without error**, but they have no effect
- **All access falls back to private** - only the object owner can access
- **Failure mode is restrictive (safe)**: Setting `public-read` does NOT make data public; unauthorized users get `AccessDenied`
- _Impact: Applications relying on ACLs to share data publicly or cross-account will fail; however, private data is never accidentally exposed_

**Bucket Policy Cross-Account Access**

- IAM `CreateAccessKey` operation not supported
- Bucket policies for cross-account access not functional

**CORS (Cross-Origin Resource Sharing)**

- No CORS configuration returned (`NoSuchCORSConfiguration`)
- Preflight (OPTIONS) requests return HTTP 403
- Response headers lack `Access-Control-Allow-Origin`
- _Impact: Browser-based applications cannot access S3 directly_

**Affected tests**:

- `test_object_acl_canned_bucketownerfullcontrol`

**1. Presigned GET/DELETE/GetObjectRetention Return 401 Unauthorized**

The presigned URL is generated correctly, but execution fails with HTTP 401:

- `execute_getObject` - HTTP 401 Unauthorized
- `execute_deleteObject` - HTTP 401 Unauthorized
- `execute_getObjectRetention` - HTTP 401 Unauthorized

This suggests that presigned URL signature verification fails for GET/DELETE operations.

**2. CORS Preflight Requests Fail**

All preflight (OPTIONS) requests return HTTP 403:

- `preflight_putObject` - 403, missing headers
- `preflight_deleteObject` - 403, missing headers

Response headers show:

- `allow_origin`: empty
- `allow_methods`: empty
- `allow_headers`: empty

**3. CORS Response Headers Missing**

All responses lack required CORS headers:

- `Access-Control-Allow-Origin` not present
- `Access-Control-Expose-Headers` not present

This means browsers cannot access:

- ETag, Content-Length, Content-Type, Last-Modified headers
- Custom headers like `x-fil-cid`

## Top 5 Gaps Most Likely to Impact Users

### 1. ACLs Ignored (Fails Restrictive/Safe)

**User Impact**: Applications that set ACLs like `public-read` or `bucket-owner-full-control` will find they have no effect:

- Setting `public-read` on an object does NOT make it publicly accessible
- Setting `bucket-owner-full-control` does NOT grant cross-account access
- All objects remain private (owner-only access) regardless of ACL setting
- **This is a safe failure**: data configured as "public" stays private, not the other way around

### 2. Presigned GET/DELETE URLs Return 401

**User Impact**: Common workflows that fail:

- Sharing download links with external users
- Pre-signed URLs for time-limited downloads
- Browser-based file downloads
- Mobile apps downloading via presigned URLs
- Any client that deletes objects via presigned URLs

### 3. CORS Not Configured

**User Impact**: Browser-based applications cannot use FTH S3:

- Single-page applications (React, Vue, Angular)
- Direct browser uploads/downloads
- Web-based file managers
- Any JavaScript S3 client (AWS SDK for JavaScript)

### 4. Metadata Key Case Conversion

**User Impact**: Applications may fail silently:

- Applications checking for specific metadata keys by name
- Migration from AWS S3 where lowercase keys were used
- Interoperability with tools that expect lowercase metadata
- Case-sensitive metadata comparisons in application logic

### 5. Cross-Account Access Not Supported

**User Impact**: Multi-tenant and enterprise scenarios fail:

- Data sharing between AWS accounts
- Service-to-service access via bucket policies
- IAM role assumption for cross-account access
- Centralized data lakes accessed by multiple accounts

## Recommendations

1. **For public access workflows**: ACLs won't work; use presigned URLs instead (note: presigned GET currently broken)
2. **For browser applications**: Configure CORS out-of-band with FTH or use a backend proxy
3. **For metadata**: Use Title-Case keys (e.g., `x-amz-meta-Filename` instead of `x-amz-meta-filename`)
4. **For presigned downloads**: Currently blocked; investigate signature verification for GET operations
5. **For cross-account access**: Use direct IAM credentials instead of bucket policies
